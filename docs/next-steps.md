# LiarCard 下一步开发指南

## 1. 当前项目状态

当前项目已经完成了本地单机模式的最小基础骨架：

- `Card` 定义了花色、点数和牌 ID。
- `PlayerState` 可以保存玩家 ID 和手牌。
- `GameState` 可以保存玩家列表、中央牌堆、当前玩家和当前阶段。
- `RoundState` 可以保存一轮的起始玩家、声明点数、上一名出牌者、上一手牌、不跟记录和待确认胜利玩家。
- `DeckManager` 已经可以创建牌堆、洗牌、发牌和排序手牌。
- `LocalGameService.StartGame()` 已经可以创建玩家、洗牌、发牌，并初始化第一轮。
- `GameBootstrap` 已经可以在 Unity 启动时创建本地游戏服务并开始游戏。

也就是说，项目目前可以完成“开始游戏并发牌”，但还没有形成完整的一局游戏。

当前还缺少这些核心能力：

- 玩家出牌。
- 玩家不跟。
- 玩家质疑。
- 中央牌堆结算。
- 回合推进。
- 胜利判定。
- 可视化调试输出。
- UI 操作入口。

近期目标应该是先完成本地单机规则闭环，而不是马上做联网、AI 或完整 UI。

## 2. 下一阶段目标

下一阶段的目标是：

> 在没有正式 UI 的情况下，让 `LocalGameService` 可以完整跑通一局本地规则。

完成后应当可以通过代码调用模拟以下流程：

1. 创建 4 名玩家。
2. 发牌。
3. 当前玩家声明点数并出若干张牌。
4. 下一个玩家选择跟牌、不跟或质疑。
5. 质疑后根据上一手真实牌面结算中央牌堆。
6. 所有人不跟时清空中央牌堆，并由最后出牌者开启新一轮。
7. 玩家出完手牌后进入待质疑窗口。
8. 如果没人成功质疑，则该玩家获胜。

## 3. 推荐实现顺序

### 第一步：补齐基础状态操作

优先补充状态类本身需要的基础方法，让服务层不要直接散乱修改内部字段。

建议修改文件：

- `Client/Assets/Scripts/LocalGame/Core/PlayerState.cs`
- `Client/Assets/Scripts/LocalGame/Core/RoundState.cs`
- `Client/Assets/Scripts/LocalGame/Core/GameState.cs`

建议新增能力：

- `PlayerState.RemoveCardsFromHand(...)`
- `PlayerState.AddCardsToHand(...)`
- `RoundState.RecordPlay(...)`
- `RoundState.RecordPass(...)`
- `RoundState.HasLastPlay`
- `GameState.MoveToPlayer(...)`
- `GameState.MoveToNextPlayer(...)`

验收标准：

- 玩家手牌可以安全增加和移除。
- 回合状态可以记录上一手出牌。
- 回合状态可以记录不跟玩家。
- 当前玩家可以稳定切换。

### 第二步：实现出牌

在 `LocalGameService` 中实现出牌动作，这是后续不跟和质疑的前提。

建议新增方法：

```csharp
public void PlayCards(List<Card> cards, Rank declaredRank)
```

出牌时需要处理：

- 游戏必须已经开始。
- 当前玩家必须拥有这些牌。
- 第一名出牌者可以声明任意点数。
- 同一轮后续玩家必须沿用当前声明点数。
- 出牌后从玩家手牌移除这些牌。
- 出牌后把牌加入 `CenterPile`。
- 出牌后记录 `LastPlayedPlayerId` 和 `LastPlayedCards`。
- 成功出牌后清空本轮的不跟记录。
- 如果玩家手牌为空，进入 `PendingWin` 或 `WaitForChallenge` 阶段。
- 否则切到下一个玩家。

验收标准：

- 出牌后玩家手牌数量减少。
- 中央牌堆数量增加。
- `RoundState.LastPlayedCards` 等于刚出的牌。
- 后续玩家不能随意改变本轮声明点数。

### 第三步：实现不跟

实现玩家跳过当前行动。

建议新增方法：

```csharp
public void Pass()
```

不跟时需要处理：

- 当前玩家不能是上一名成功出牌者。
- 记录当前玩家已不跟。
- 如果还没有所有其他玩家都不跟，则切到下一个可行动玩家。
- 如果所有其他玩家都不跟，则清空中央牌堆。
- 清空中央牌堆后，由最后出牌者开启新一轮。

验收标准：

- 玩家不跟后不会立刻改变中央牌堆。
- 所有其他玩家都不跟后，中央牌堆被清空。
- 新一轮起始玩家是上一名成功出牌者。

### 第四步：实现质疑

在出牌和中央牌堆都可靠之后，再实现质疑。

建议新增方法：

```csharp
public ChallengeResult Challenge()
```

质疑规则：

- 只能质疑上一名出牌者。
- 只检查上一名出牌者刚出的牌。
- 如果上一手牌全部是声明点数或 Joker，则质疑失败。
- 如果上一手牌中存在不符合声明点数的普通牌，则质疑成功。
- 质疑失败时，质疑者拿走中央牌堆。
- 质疑成功时，上一名出牌者拿走中央牌堆。
- 质疑结算后，由质疑胜利方开启新一轮。

验收标准：

- 真牌被质疑时，质疑者收走中央牌堆。
- 假牌被质疑时，出牌者收走中央牌堆。
- 结算后中央牌堆为空。
- 新一轮起始玩家正确。

### 第五步：实现胜利判定

胜利判定不要和普通出牌混在一起直接结束游戏，应当先进入待质疑状态。

建议规则：

- 玩家打出最后一张牌后，先进入 `PendingWin`。
- 其他玩家有机会质疑最后一手。
- 如果质疑成功，该玩家拿走中央牌堆，游戏继续。
- 如果没人质疑成功，该玩家获胜，进入 `GameOver`。

建议新增状态：

- `WinnerPlayerId`
- `PendingWinnerPlayerId`

验收标准：

- 玩家出完牌后不会立刻无条件胜利。
- 最后一手真牌通过质疑窗口后才胜利。
- 最后一手假牌被成功质疑后游戏继续。

## 4. 暂时不要急着做的事情

下面这些内容很重要，但不建议现在就做：

- 正式 UI。
- 3D 牌面表现。
- Go 服务端。
- 在线同步。
- AI 训练。
- 职业技能系统。
- 测谎仪概率系统。

原因是本地规则还没有闭环。现在做 UI 或联网，会让问题分散到表现层和通信层，调试成本会变高。

## 5. 建议的调试方式

在正式 UI 出来前，可以先做一个很小的调试入口。

建议在 `GameBootstrap` 中临时输出：

- 玩家数量。
- 每名玩家手牌数量。
- 当前玩家。
- 当前阶段。
- 中央牌堆数量。
- 当前声明点数。
- 上一名出牌者。

示例调试目标：

```text
StartGame 后：
Player 1: 14 cards
Player 2: 14 cards
Player 3: 13 cards
Player 4: 13 cards
CurrentPlayer: Player 1
Phase: RoundStart
CenterPile: 0
```

后续每实现一个动作，都先用 `Debug.Log` 验证状态是否正确。

## 6. 建议的提交节奏

每完成一个小闭环就提交一次，不要等所有规则都写完。

推荐提交顺序：

1. `补齐本地游戏状态操作`
2. `实现本地出牌逻辑`
3. `实现本地不跟逻辑`
4. `实现本地质疑结算`
5. `实现本地胜利判定`
6. `添加本地游戏调试输出`

每次提交前建议运行：

```powershell
dotnet build Client\Assembly-CSharp.csproj
git status --short
```

如果 Unity 自动修改了场景或项目设置，需要确认这些改动是否确实属于本次任务，再决定是否一起提交。

## 7. 完成本阶段的定义

当下面条件全部满足时，可以认为本地单机规则第一版完成：

- 可以开始游戏并发牌。
- 可以出牌。
- 可以不跟。
- 可以质疑。
- 可以处理中央牌堆归属。
- 可以开启新一轮。
- 可以判断胜利。
- 可以通过日志看清楚每一步状态变化。
- 不依赖 UI 也能模拟一局游戏流程。

完成这个阶段后，再进入 UI 表现层会顺很多：UI 只需要调用已经稳定的 `LocalGameService`，而不是一边做界面一边修规则。
